## What This Skill Does

Prepares concise, targeted briefing materials for meetings with specific stakeholders. Adapts content, framing, and emphasis based on who the audience is and what they care about.

## When to Use

- User has an upcoming meeting with a key stakeholder
- User needs to present to BTG PE partners, board, or cross-entity leaders
- User needs talking points for a difficult conversation
- User is preparing for a review or strategic session

## Methodology

### 1. Understand the Meeting
- **Who:** Specific stakeholder(s) and their known priorities
- **What:** Topic, agenda, desired outcome
- **Context:** History of this relationship/topic, any sensitivities
- **Stakes:** What does the user want to walk out with?

### 2. Build the Briefing

**Format — One-Page Briefing:**
```
MEETING BRIEF: [Topic]
Date: [Date] | With: [Stakeholder(s)]

CONTEXT (2-3 sentences)
What's this about and why now?

KEY POINTS (3-5 bullets)
What you want to communicate

THEIR LIKELY CONCERNS
What they'll push back on or ask about

YOUR ASK
What you need from them

LANDMINES
Topics to avoid or handle carefully

SUGGESTED OPENING
How to frame the conversation
```

### 3. Stakeholder-Specific Adaptation

| Audience | Emphasis | Tone |
|----------|----------|------|
| BTG PE partners | Financial returns, timeline, risk | Data-driven, concise |
| Board | Strategic direction, governance | Authoritative, structured |
| Peer leaders (other directors) | Collaboration, mutual benefit | Collegial, win-win |
| Technical teams | Feasibility, implementation | Specific, detailed |
| External partners | Value proposition, alignment | Professional, compelling |

### 4. Red Team
- What's the hardest question they could ask?
- Prepare 3 likely questions with recommended answers
- Identify if there's something the user should bring up first (before it's asked)

## Pitfalls to Avoid
- Generic briefings that could apply to anyone — each should feel tailored
- Ignoring political dynamics between stakeholder and user
- Overloading with information when the meeting is about one decision
- Forgetting to identify the "ask" — every meeting should have a clear objective

## Advancement Criteria
- **→ L1:** First briefing created and used
- **→ L2:** User provides feedback on format preferences; briefings start reflecting specific stakeholder profiles stored in USER.md
- **→ L3:** Briefings consistently anticipate the right questions; user requests them proactively before important meetings
- **→ L4:** Maintained stakeholder profiles enable nuanced framing; post-meeting debriefs feed back into stakeholder intelligence
