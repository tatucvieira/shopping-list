## What This Skill Does

Transforms ambiguous situations into clear, decidable frames. The most valuable thing a Chief of Staff can do is not make decisions — it's make decisions **easier to make well**.

## When to Use

- User is wrestling with a complex decision
- Multiple stakeholders have conflicting views and user needs to navigate
- There's a "boiling frog" situation where no one is making a call
- User says things like "I don't know what to do about X" or "this is a mess"

## Methodology

### 1. Name the Decision
Most "messy situations" are actually 2-3 separate decisions tangled together. Untangle them.

**Template:**
> "The core decision is: [VERB] [SPECIFIC THING] by [TIMEFRAME], given [KEY CONSTRAINT]."

Example: "The core decision is: whether to build a unified CX platform for Nio and V.tal by Q3, given that both teams currently operate separate systems."

### 2. Identify the Decision Type

| Type | Description | Approach |
|------|-------------|----------|
| **One-way door** | Irreversible or very costly to reverse | High scrutiny, more analysis |
| **Two-way door** | Reversible, can course-correct | Bias toward action, decide fast |
| **Sequencing** | Order matters more than choice | Map dependencies first |
| **Portfolio** | Multiple bets, not one big call | Diversify, set kill criteria |

### 3. Map the Decision Space

**Decision Matrix:**
| Option | [Criteria 1] | [Criteria 2] | [Criteria 3] | Score |
|--------|-------------|-------------|-------------|-------|
| Option A | ... | ... | ... | ... |
| Option B | ... | ... | ... | ... |
| Option C (incl. "do nothing") | ... | ... | ... | ... |

**Criteria should be:**
- Weighted by what actually matters (not equal weights by default)
- Specific enough to differentiate options
- Aligned with the user's stated priorities

### 4. Surface the Real Constraint

Usually the decision is hard not because of the options, but because of a hidden constraint:
- Political: "We can't do X because person Y would block it"
- Resource: "We don't have the team for the best option"
- Information: "We don't know enough to choose confidently"
- Values: "Both options conflict with something we care about"

Name the constraint explicitly. Often, solving the constraint makes the decision obvious.

### 5. Deliver the Frame

**Template:**
```
DECISION: [Clear statement]
TYPE: [One-way / Two-way / Sequencing / Portfolio]
DEADLINE: [When this must be decided]
REAL CONSTRAINT: [What actually makes this hard]

OPTIONS:
1. [Option] — [One-line description]
   ✓ Pros: ...
   ✗ Cons: ...
   ⚠ Risk: ...

2. [Option] ...

3. [Option] ...

MY RECOMMENDATION: [Option N]
CONFIDENCE: [High/Medium/Low]
BECAUSE: [2-3 sentences on why]

IF I'M WRONG: [What would need to be true for a different option to win]
```

## Pitfalls to Avoid
- Presenting options without a recommendation (that's delegation, not framing)
- Ignoring the political dimension of decisions
- Creating false urgency or false patience
- Overcomplicating simple decisions with heavy frameworks
- Treating all decisions as one-way doors

## Advancement Criteria
- **→ L1:** First decision framed and used
- **→ L2:** User adopts the framing language; frames become tailored to holding context
- **→ L3:** Can frame multi-entity decisions that balance V.tal/Nio/Tecto trade-offs
- **→ L4:** Decision tracking shows improved outcomes; user relies on framing for all major calls
